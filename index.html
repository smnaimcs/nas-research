Neural Architecture Search, priced in FLOPs, memory, and milliseconds.

This book is a working plan for a single research thread: architectures that are searched for, compressed, and deployed with edge–cloud resource limits treated as first‑class objectives — not applied afterward as an afterthought. Every phase below pairs the concepts worth reading about with the code that turns them into a running system.

The plan is organized the way the system itself is organized: start with a search algorithm that knows nothing about resource limits, then teach it about compression, then about the two-tier world it will actually run in, then about the tradeoffs between all of those things at once. By the last phase the same object — an architecture — carries an accuracy score, a compression ratio, an edge‑tier cost, a cloud‑tier cost, and a user‑facing quality estimate, all computed by pieces built in earlier steps.

Phase 01 · Foundation Knowledge Building

1 · Understanding NAS Fundamentals

search space → search strategy → performance estimation

Neural Architecture Search automates the part of model design that used to be manual trial and error: choosing depth, width, and operator types. Every NAS method, however sophisticated, is answering three questions — what architectures are allowed (the search space), how the space gets explored (the search strategy), and how a candidate gets scored without paying the full cost of training it (the performance estimation strategy). Random search is the right place to start precisely because it isolates those three pieces without an optimizer's machinery in the way.

Read first Neural Architecture SearchAutoML Evolutionary algorithmsReinforcement learning
Key concepts
  • Search space — the set of architectures that can be represented at all
  • Search strategy — how the space is explored (random, evolutionary, RL, gradient‑based)
  • Performance estimation — how a candidate is scored cheaply (proxy accuracy, early stopping, weight sharing)
simple_nas.py
# Simple random NAS on MNIST
import random
import tensorflow as tf

class SimpleNAS:
    def __init__(self):
        self.search_space = {
            'num_layers': [2, 3, 4],
            'filters': [32, 64, 128],
            'kernel_size': [3, 5]
        }

    def random_search(self, num_samples=10):
        best_arch = None
        best_acc = 0

        for i in range(num_samples):
            arch = {
                'num_layers': random.choice(self.search_space['num_layers']),
                'filters': random.choice(self.search_space['filters']),
                'kernel_size': random.choice(self.search_space['kernel_size'])
            }
            acc = self.evaluate_architecture(arch)
            if acc > best_acc:
                best_acc = acc
                best_arch = arch

        return best_arch, best_acc
Why start here

Random search has no hyperparameters to tune and no failure modes to debug. It exists to validate the evaluation loop — evaluate_architecture — that every later, smarter search strategy will reuse unchanged.

Phase 01 · Foundation Knowledge Building

2 · Model Compression Techniques

pruning → quantization → distillation

Compression is what turns a model that scores well into a model that fits. Pruning removes weights or whole channels judged unimportant by some criterion (here, L1 norm). Quantization reduces numeric precision — often from 32‑bit floats to 8‑bit integers — trading a small accuracy cost for large memory and latency wins. Knowledge distillation trains a small "student" network to mimic a large "teacher," transferring behavior rather than weights. None of these are exclusive; the later compression‑aware search step treats them as a menu of operators applied to a candidate architecture.

# Channel pruning example
class ChannelPruning:
    def prune_channels(self, model, pruning_ratio=0.3):
        """Prune channels based on L1 norm of weights"""
        pruned_model = model.clone()
        for layer in pruned_model.conv_layers:
            # Calculate importance scores
            importance = torch.norm(layer.weight, p=1, dim=[1,2,3])
            # Keep top-k channels
            k = int(len(importance) * (1 - pruning_ratio))
            top_channels = torch.topk(importance, k).indices
            layer.weight = layer.weight[top_channels]
        return pruned_model

# Quantization example
def apply_quantization(model, bits=8):
    """Simple uniform quantization"""
    import torch.quantization as quant

    model.qconfig = quant.get_default_qconfig('fbgemm')
    quant.prepare(model, inplace=True)
    quant.convert(model, inplace=True)
    return model

Phase 02 · Edge–Cloud Constraints Integration

3 · Understanding Edge–Cloud Systems

two tiers, two budgets

An edge device and a cloud server are not the same computer with different specs — they represent different budgets for the same four resources: compute (FLOPs), memory, latency, and energy. Modeling both tiers explicitly, rather than assuming "cloud‑sized" compute everywhere, is what makes later search results deployable rather than theoretical. The constraint check below is deliberately simple: a candidate architecture either fits a tier's budget or it doesn't.

class EdgeCloudConstraints:
    def __init__(self):
        self.constraints = {
            'edge': {
                'max_flops': 1e9,       # 1 GFLOP
                'max_memory': 512e6,    # 512 MB
                'max_latency': 10e-3,   # 10 ms
                'energy_budget': 1.0    # 1 Joule
            },
            'cloud': {
                'max_flops': 100e9,     # 100 GFLOP
                'max_memory': 16e9,     # 16 GB
                'max_latency': 100e-3,  # 100 ms
                'energy_budget': 100.0  # 100 Joules
            }
        }

    def check_viability(self, architecture, tier='edge'):
        """Check if architecture meets tier constraints"""
        arch_cost = self.estimate_cost(architecture)
        constraints = self.constraints[tier]

        return (arch_cost['flops'] <= constraints['max_flops'] and
                arch_cost['memory'] <= constraints['max_memory'] and
                arch_cost['latency'] <= constraints['max_latency'])

Phase 02 · Edge–Cloud Constraints Integration

4 · Multi‑Objective Optimization

accuracy vs. flops vs. memory vs. latency

Once accuracy is one objective among several, "best" stops being a single number. Pareto dominance gives a rigorous replacement: architecture A dominates B only if A is at least as good on every objective and strictly better on one. The set of non‑dominated solutions — the Pareto front — is the honest output of a multi‑objective search: a menu of real tradeoffs rather than one winner.

Read first Pareto optimalityMulti-objective optimization NSGA-II algorithmConstraint satisfaction problems
pareto.py
class MultiObjectiveNAS:
    def __init__(self):
        self.objectives = ['accuracy', 'flops', 'memory', 'latency']

    def pareto_dominance(self, solution_a, solution_b):
        """Check if solution_a dominates solution_b"""
        # We want to maximize accuracy, minimize others
        better_in_one = False

        # Check accuracy (higher is better)
        if solution_a['accuracy'] > solution_b['accuracy']:
            better_in_one = True
        elif solution_a['accuracy'] < solution_b['accuracy']:
            return False

        # Check costs (lower is better)
        for obj in ['flops', 'memory', 'latency']:
            if solution_a[obj] > solution_b[obj]:
                return False
            elif solution_a[obj] < solution_b[obj]:
                better_in_one = True

        return better_in_one

    def find_pareto_front(self, solutions):
        """Extract Pareto-optimal solutions"""
        pareto_front = []
        for i, sol_a in enumerate(solutions):
            dominated = False
            for j, sol_b in enumerate(solutions):
                if i != j and self.pareto_dominance(sol_b, sol_a):
                    dominated = True
                    break
            if not dominated:
                pareto_front.append(sol_a)
        return pareto_front

Phase 03 · Distributed System Simulation

5 · Building the Simulation Framework

discrete-event simulation of edge nodes, cloud, and the network between them

Real edge fleets are expensive to prototype against. A discrete‑event simulator built on simpy stands in: edge nodes queue and process requests with a single‑slot resource, a network model injects realistic transfer latency between tiers, and a metrics collector accumulates everything the evaluation phase will later need. This is the substrate every subsequent phase runs on top of.

import simpy
import numpy as np

class EdgeCloudSimulator:
    def __init__(self):
        self.env = simpy.Environment()
        self.edge_nodes = []
        self.cloud_resources = []
        self.network = NetworkModel()
        self.metrics_collector = MetricsCollector()

    def add_edge_node(self, compute_power, memory, energy):
        node = EdgeNode(self.env, compute_power, memory, energy)
        self.edge_nodes.append(node)
        return node

    def run_simulation(self, duration):
        self.env.run(until=duration)
        return self.metrics_collector.get_summary()

class EdgeNode:
    def __init__(self, env, compute_power, memory, energy):
        self.env = env
        self.compute_power = compute_power  # FLOPS
        self.memory = memory                # bytes
        self.energy = energy                # Watts
        self.queue = simpy.Resource(env, capacity=1)
        self.metrics = []

    def process_request(self, request):
        with self.queue.request() as req:
            yield req
            # Simulate processing time
            processing_time = self.estimate_time(request)
            yield self.env.timeout(processing_time)
            return self.execute(request)

class NetworkModel:
    def __init__(self):
        self.edge_to_edge_latency = (5e-3, 10e-3)      # 5-10ms
        self.edge_to_cloud_latency = (50e-3, 100e-3)   # 50-100ms
        self.bandwidth = 100e6  # 100 Mbps

    def transfer_data(self, data_size):
        latency = np.random.uniform(*self.edge_to_cloud_latency)
        transfer_time = data_size / self.bandwidth
        return latency + transfer_time

Phase 03 · Distributed System Simulation

6 · NAS with Compression-Awareness

where search, compression, and simulation meet

This is the hinge of the whole system. Every candidate architecture is now run through each compression operator from Step 2, and each compressed variant is evaluated on both simulated tiers from Step 5. The result is a table of architecture × compression × tier combinations, each with its own Pareto score — a single scalar that blends accuracy, resource efficiency, latency, and energy so that candidates can be ranked without discarding the underlying multi‑objective detail.

compression_aware_nas.py
class CompressionAwareNAS:
    def __init__(self, simulator):
        self.simulator = simulator
        self.compression_operators = [
            ChannelPruning(),
            WeightPruning(),
            Quantization(bits=8),
            KnowledgeDistillation()
        ]

    def evaluate_architecture_with_compression(self, architecture):
        """Evaluate architecture-performance-compression tradeoffs"""
        results = []

        # Base model evaluation
        base_metrics = self.evaluate_base_model(architecture)

        for compression_op in self.compression_operators:
            compressed_model = compression_op.apply(architecture)

            # Evaluate on edge
            edge_metrics = self.simulator.simulate_execution(
                compressed_model,
                tier='edge'
            )

            # Evaluate on cloud
            cloud_metrics = self.simulator.simulate_execution(
                compressed_model,
                tier='cloud'
            )

            results.append({
                'architecture': architecture,
                'compression': compression_op.name,
                'compression_ratio': compression_op.compression_ratio,
                'edge_metrics': edge_metrics,
                'cloud_metrics': cloud_metrics,
                'pareto_score': self.calculate_pareto_score(edge_metrics, cloud_metrics)
            })

        return results

    def calculate_pareto_score(self, edge_metrics, cloud_metrics):
        """Multi-objective scoring considering both edge and cloud performance"""
        score = 0
        # Accuracy maintained
        score += edge_metrics['accuracy'] * 0.4
        # Resource efficiency on edge
        score += (1 - edge_metrics['resource_usage']) * 0.3
        # Latency optimization
        score += (1 - edge_metrics['latency'] / edge_metrics['latency_budget']) * 0.2
        # Energy efficiency
        score += (1 - edge_metrics['energy'] / edge_metrics['energy_budget']) * 0.1
        return score
Weighting note

The 0.4 / 0.3 / 0.2 / 0.1 split above is a starting default, not a finding — treat it as the first knob to sensitivity‑test once the pipeline runs end to end.

Phase 04 · Adaptation Mechanisms

7 · Dynamic Adaptation System

picking a model at runtime, not just at design time

A search process picks architectures once, offline. A deployed system faces conditions that change minute to minute — battery draining, bandwidth dropping, memory pressure rising. The adaptation layer closes that gap: a context monitor samples device and network state, and a decision engine maps that state onto one of several pre‑registered models (ultra‑low‑power, memory‑efficient, edge‑heavy, balanced) rather than trying to re‑run search live.

adaptation.py
class AdaptiveDeployment:
    def __init__(self):
        self.context_monitor = ContextMonitor()
        self.model_registry = ModelRegistry()
        self.decision_engine = DecisionEngine()

    def adapt_to_context(self, current_context):
        """Dynamically select/adapt model based on context"""
        context = self.context_monitor.get_context()

        # Check resource availability
        available_resources = {
            'edge_compute': context['edge_cpu_available'],
            'edge_memory': context['edge_memory_available'],
            'bandwidth': context['network_bandwidth'],
            'battery': context['battery_level']
        }

        # Decision logic for model selection
        if available_resources['battery'] < 0.2:
            return self.model_registry.get_ultra_low_power_model()
        elif available_resources['edge_memory'] < 100e6:
            return self.model_registry.get_memory_efficient_model()
        elif available_resources['bandwidth'] < 1e6:
            return self.model_registry.get_edge_heavy_model()
        else:
            return self.model_registry.get_balanced_model()

class ContextMonitor:
    def __init__(self):
        self.metrics_history = []

    def get_context(self):
        return {
            'edge_cpu_available': self.measure_cpu_availability(),
            'edge_memory_available': self.measure_memory_availability(),
            'network_bandwidth': self.measure_bandwidth(),
            'battery_level': self.measure_battery(),
            'workload_intensity': self.measure_workload(),
            'data_distribution': self.measure_data_drift()
        }

Phase 05 · Integrated System and Evaluation

8 · Putting It All Together

the end-to-end experiment loop

Everything from Steps 1–7 becomes a single orchestrator here: set up the simulated environment, run compression‑aware NAS under constraints, then evaluate each discovered architecture two ways — as a static deployment and as an adaptively‑deployed one — so the report can quantify exactly how much the adaptation layer from Step 7 is worth.

system.py
class EdgeCloudNASSystem:
    def __init__(self):
        self.simulator = EdgeCloudSimulator()
        self.nas_engine = CompressionAwareNAS(self.simulator)
        self.adaptation = AdaptiveDeployment()
        self.metrics = PerformanceEvaluator()

    def run_experiment(self, config):
        """End-to-end experiment execution"""
        results = []

        # Setup simulation environment
        self.setup_simulation(config['environment'])

        # Run NAS with compression constraints
        architectures = self.nas_engine.search(
            search_space=config['search_space'],
            constraints=config['constraints'],
            num_iterations=config['nas_iterations']
        )

        # Evaluate found architectures
        for arch in architectures:
            # Static evaluation
            static_metrics = self.evaluate_static_deployment(arch)

            # Adaptive evaluation
            adaptive_metrics = self.evaluate_adaptive_deployment(arch)

            results.append({
                'architecture': arch,
                'static_performance': static_metrics,
                'adaptive_performance': adaptive_metrics,
                'compression_impact': self.analyze_compression_impact(arch),
                'edge_cloud_tradeoff': self.analyze_tradeoffs(arch)
            })

        return self.generate_report(results)

class PerformanceEvaluator:
    def __init__(self):
        self.metrics = {
            'system': ['throughput', 'latency', 'scalability'],
            'model': ['flops', 'memory', 'inference_time', 'accuracy'],
            'qos': ['availability', 'reliability', 'user_satisfaction'],
            'security': ['data_breach_freq', 'encryption_strength']
        }

    def evaluate_system_performance(self, deployment_results):
        """Evaluate system-level metrics"""
        return {
            'throughput': self.calculate_throughput(deployment_results),
            'latency': self.calculate_cross_tier_latency(deployment_results),
            'scalability': self.measure_scalability(deployment_results),
            'accuracy_vs_efficiency': self.plot_pareto_frontier(deployment_results)
        }

    def calculate_cross_tier_latency(self, results):
        """Analyze latency across edge-cloud tiers"""
        edge_latency = np.mean([r['edge_latency'] for r in results])
        cloud_latency = np.mean([r['cloud_latency'] for r in results])
        transfer_latency = np.mean([r['transfer_latency'] for r in results])

        return {
            'edge_processing': edge_latency,
            'cloud_processing': cloud_latency,
            'data_transfer': transfer_latency,
            'total': edge_latency + cloud_latency + transfer_latency,
            'breakdown_percentages': {
                'edge': edge_latency / (edge_latency + cloud_latency + transfer_latency),
                'cloud': cloud_latency / (edge_latency + cloud_latency + transfer_latency),
                'transfer': transfer_latency / (edge_latency + cloud_latency + transfer_latency)
            }
        }

Phase 06 · Benchmarking and Validation

9 · Implement Standard Benchmarks

checking results against external, scenario-driven suites

Internal metrics can drift from what matters in practice, so results get cross‑checked against named benchmark suites with scenario‑specific constraints — ICU monitoring demands five‑nines reliability and HIPAA‑grade privacy at 5 ms latency; autonomous‑vehicle perception demands 1 ms latency and a fail‑safe mode. These aren't arbitrary numbers — they're the kind of hard constraint that separates a publishable system from a lab demo.

benchmarks.py
class BenchmarkSuite:
    def __init__(self):
        self.benchmarks = {
            'edge_aibench': EdgeAIBench(),
            'edgebench': EdgeBench(),
            'kheops': KheOps()
        }

    def run_benchmarks(self, system, datasets):
        results = {}

        for name, benchmark in self.benchmarks.items():
            results[name] = benchmark.evaluate(system, datasets[name])

        return self.compare_results(results)

    def compare_results(self, results):
        """Cross-benchmark comparison"""
        comparison = {}
        metrics = ['accuracy', 'latency', 'energy', 'memory']

        for metric in metrics:
            comparison[metric] = {
                bench: results[bench][metric]
                for bench in results.keys()
            }

        return comparison

class EdgeAIBench:
    def __init__(self):
        self.scenarios = ['icu_monitoring', 'autonomous_vehicles']

    def evaluate(self, system, dataset):
        results = {}
        for scenario in self.scenarios:
            # Setup scenario-specific constraints
            constraints = self.get_scenario_constraints(scenario)
            # Run evaluation
            results[scenario] = system.evaluate(dataset, constraints)
        return results

    def get_scenario_constraints(self, scenario):
        if scenario == 'icu_monitoring':
            return {
                'max_latency': 5e-3,     # 5ms for real-time monitoring
                'reliability': 0.9999,   # 99.99% uptime
                'privacy_level': 'high'  # HIPAA compliance
            }
        elif scenario == 'autonomous_vehicles':
            return {
                'max_latency': 1e-3,      # 1ms for safety-critical
                'reliability': 0.99999,   # 99.999% uptime
                'fail_safe': True
            }

Phase 07 · QoS–QoE Integration

10 · Quality of Experience Framework

from measured system metrics to felt user experience

The final layer translates system‑side Quality of Service numbers — availability, latency, reliability — into the Quality of Experience a person actually feels. An exponential decay model captures the intuition that perceived quality degrades sharply past some latency threshold rather than linearly, and correlation analysis against user‑satisfaction data locates that threshold empirically instead of guessing it.

qos_qoe.py
class QoSEvaluator:
    def __init__(self):
        self.qos_metrics = ['availability', 'reliability', 'response_time']
        self.qoe_metrics = ['user_satisfaction', 'perceived_quality', 'usability']

    def evaluate_qos_qoe_relationship(self, system_results):
        """Map QoS metrics to QoE outcomes"""
        relationships = {}

        # Availability vs User Satisfaction
        availability = system_results['availability']
        user_satisfaction = self.simulate_user_satisfaction(availability)

        # Latency vs Perceived Quality
        latency = system_results['latency']
        perceived_quality = self.model_perceived_quality(latency)

        # Reliability vs Usability
        reliability = system_results['reliability']
        usability = self.calculate_usability_score(reliability)

        return {
            'availability_satisfaction': {
                'correlation': np.corrcoef(availability, user_satisfaction)[0,1],
                'threshold': self.find_satisfaction_threshold(availability, user_satisfaction)
            },
            'latency_quality': {
                'model': self.fit_qoe_model(latency, perceived_quality),
                'acceptable_latency': self.find_acceptable_threshold(latency, perceived_quality, 0.8)
            },
            'reliability_usability': {
                'mapping_function': self.create_mapping(reliability, usability),
                'critical_points': self.find_critical_points(reliability, usability)
            }
        }

    def model_perceived_quality(self, latency):
        """QoE model: perceived quality decreases with latency"""
        # Exponential decay model for perceived quality
        base_quality = 1.0
        decay_rate = 0.1
        return base_quality * np.exp(-decay_rate * latency)
